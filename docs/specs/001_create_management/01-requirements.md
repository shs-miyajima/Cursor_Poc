# 要件定義 — 社内備品貸出管理

> status: 01-requirements.status を参照（`draft` | `approved` | `rejected`）

## 1. 概要

社員が開発用 PC や検証用端末などの社内備品を申請・使用するための、Laravel 管理画面向けの簡易貸出管理機能を作成する。PoC のためパスワード認証は実装せず、画面上のユーザー選択による擬似ログインでロール別の表示・操作を切り替える。

本機能の機能要件は外部資料 `specification.md` を原典とする。ただし、実装・テスト判断の正本は本ファイルを含む `docs/specs/001_create_management/` とし、環境・技術スタックは既存 Docker 構成を正とする。

## 2. 用語集

| 用語 | 定義 |
|------|------|
| 擬似ログイン | パスワード認証を使わず、画面上の「現在のユーザー」選択またはリクエストパラメータ `mock_user_id` により操作ユーザーを切り替える方式 |
| 管理者 | `users.role = admin` のユーザー。全ユーザーの申請閲覧とステータス更新が可能 |
| 一般社員 | `users.role = staff` のユーザー。自分の申請閲覧と新規申請のみ可能 |
| 申請中 | `equipment_loan_requests.status = pending`。管理者承認待ちの状態 |
| 貸出中 | `equipment_loan_requests.status = approved`。承認済みで貸出中の状態 |
| 返却済 | `equipment_loan_requests.status = returned`。返却完了した状態 |
| 却下 | `equipment_loan_requests.status = rejected`。管理者により申請が却下された状態 |
| 返却期限超過 | `status = approved` かつ `requested_to` が本日より前で、返却済になっていない状態 |

## 3. スコープ

### 対象

- [x] Laravel 管理画面
- [x] フロント（Blade / Vite / JavaScript / Tailwind CSS）
- [x] DB スキーマ変更
- [x] 備品貸出申請の一覧、検索、新規申請、ステータス更新
- [x] 返却期限超過のアラート表示と一覧行での注意喚起
- [x] 擬似ログインによるユーザー切り替え
- [x] ロールに応じたデータ閲覧・操作制御
- [x] 部署制限と重複期間の申請禁止

### 対象外

- パスワード認証、Laravel Breeze 等による通常ログイン
- 実在の社員マスタや社内認証基盤との連携
- 備品マスタの登録・編集・削除画面
- ユーザーの登録・編集・削除画面
- メール通知、承認依頼通知
- 備品マスタの登録・編集・削除画面での在庫数変更

## 4. ユーザーストーリー / ユースケース

| ID | 操作者 | 操作 | 期待結果 |
|----|--------|------|----------|
| UC-01 | 一般社員 | 現在のユーザーを自分に切り替える | 自分の申請一覧のみが表示され、管理者用ボタンは表示されない |
| UC-02 | 一般社員 | 備品、貸出開始日、貸出終了日、理由を入力して新規申請する | 部署制限、重複期間チェック、在庫数チェックを通過した場合、`pending` で申請が登録される |
| UC-03 | 一般社員 | 自部署対象外の備品を申請する | 申請は登録されず、HTTP 403 相当のエラーが表示される |
| UC-04 | 一般社員 | 既存の `pending` または `approved` 申請と期間が重なる別備品を申請する | 申請は登録されず、重複期間エラーが表示される |
| UC-05 | 管理者 | 現在のユーザーを管理者に切り替える | 全ユーザーの申請一覧が表示され、ステータス更新ボタンが表示される |
| UC-06 | 管理者 | 申請を承認・却下・返却完了に更新する | 対象申請のステータスが指定値に更新され、一覧表示に反映される |
| UC-07 | 利用者 | 備品名で検索する | 備品名の部分一致に該当する申請のみが表示される |
| UC-08 | 利用者 | 返却期限を過ぎた貸出中申請がある状態で画面を表示する | TOP 画面上部に期限切れアラートが表示され、対象レコードに注意アイコンが表示される |

## 5. 画面・API 仕様

### 5.1 画面

| 画面名 | URL | 概要 |
|--------|-----|------|
| 備品貸出管理画面 | `/equipment-loans` | 擬似ログイン切り替え、申請履歴一覧、備品名検索、新規申請、管理者向けステータス更新を行う単一画面 |

画面最上部に「現在のユーザー」を選択するドロップダウンを常時配置する。ドロップダウンを切り替えると API の再呼び出し、またはページリロードにより、一覧データと操作権限表示を選択ユーザーに合わせて更新する。

メイン画面には申請履歴をテーブル形式で表示し、ステータスはバッジで色分けする。管理者で閲覧している場合のみ、各行に「承認」「却下」「返却」のステータス変更ボタンを表示する。

返却期限超過の申請がある場合は、画面上部にアラートを表示する。一般社員では自分の返却期限超過申請のみ、管理者では全ユーザーの返却期限超過申請を対象とする。申請一覧の対象レコードには注意アイコンを表示し、通常行より視認しやすくする。

新規申請ボタンから、備品、貸出開始日、貸出終了日、理由を入力するフォームを表示する。理由は `equipment_loan_requests.reason` に任意項目として保存する。

### 5.2 画面項目定義

| 項目名 | 型 | 必須 | 最大長 | 初期値 | 表示条件・備考 |
|--------|----|------|--------|--------|----------------|
| 現在のユーザー | select | 必須 | - | 最初の一般社員 | 全ユーザーから選択。選択値は `mock_user_id` として API に送信 |
| 備品名検索 | text | 任意 | TBD | 空 | `GET /api/equipment-loans?search=...` に使用。部分一致 |
| 申請一覧: 申請者 | text | - | - | - | 管理者は全ユーザー分、一般社員は自分の申請のみ |
| 申請一覧: 備品名 | text | - | - | - | `equipment.name` |
| 申請一覧: ステータス | badge | - | - | - | `pending` / `approved` / `returned` / `rejected` を色分け |
| 申請一覧: 貸出開始日 | date | - | - | - | `requested_from` |
| 申請一覧: 貸出終了日 | date | - | - | - | `requested_to` |
| 申請一覧: 期限切れ注意 | icon | - | - | - | 返却期限超過のレコードのみ表示 |
| 画面上部: 期限切れアラート | alert | - | - | 非表示 | 返却期限超過が 1 件以上ある場合のみ表示。件数と対象概要を表示 |
| 新規申請: 備品 | select | 必須 | - | 未選択 | `equipment_id` |
| 新規申請: 貸出開始日 | date | 必須 | - | 空 | `requested_from` |
| 新規申請: 貸出終了日 | date | 必須 | - | 空 | `requested_to` |
| 新規申請: 理由 | textarea | 任意 | TBD | 空 | `reason` として保存 |
| ステータス更新ボタン | button | - | - | - | 管理者のみ表示。「承認」「却下」「返却」 |

### 5.3 API / ルート

| メソッド | パス | パラメータ | 概要 |
|----------|------|------------|------|
| GET | `/api/equipment-loans` | `mock_user_id`, `search`（任意） | 選択ユーザーのロールに応じた申請一覧を返却する。`user` と `equipment` を Eager Loading し、備品名の部分一致検索を受け付ける。返却期限超過の判定結果を一覧行とアラート表示に利用できる形で返却する |
| POST | `/api/equipment-loans` | `mock_user_id`, `equipment_id`, `requested_from`, `requested_to`, `reason`（任意） | 部署制限、重複期間チェック、在庫数チェックを行い、初期ステータス `pending` で申請を登録する |
| PATCH | `/api/equipment-loans/{id}/status` | `mock_user_id`, `status` | 管理者のみ、対象申請のステータスを `approved` / `rejected` / `returned` に更新する |

画面表示用の Web ルートは `/equipment-loans` とする。

### 5.4 フロント連携

- Blade で単一画面を表示する。
- Vite 管理の JavaScript から axios で API を呼び出す。
- `mock_user_id` はユーザー選択値から取得し、一覧取得・新規申請・ステータス更新 API に送信する。
- ユーザー切り替え時は一覧データ、管理者ボタン表示、新規申請フォームの権限表示を更新する。
- 一覧データに含まれる返却期限超過情報をもとに、画面上部アラートと対象行の注意アイコンを表示する。
- API エラーは画面上にユーザーが認識できる形で表示する。

## 6. データ要件

### 6.1 環境・DB 前提

外部資料では SQLite / Laravel 11.x が前提とされているが、本プロジェクトでは既存 Docker 構成を正とする。

| 項目 | 正とする内容 | 参照 |
|------|--------------|------|
| PHP / Laravel | Laravel 12 / PHP 8.4（Docker イメージ） | `AGENTS.md`, `composer.json`, `docker-compose.yml` |
| DB | PostgreSQL 18 | `docker-compose.yml` |
| 起動 | `run_debug.bat` → http://localhost:8000 | `AGENTS.md` |
| フロント | Blade + Vite + axios + JavaScript + Tailwind CSS | `AGENTS.md` |

### 6.2 テーブル

| テーブル | 操作 | 概要 |
|----------|------|------|
| `users` | 作成 / 参照 | 擬似ログイン対象ユーザー。氏名、部署、ロールを持つ |
| `equipments` | 作成 / 参照 | 貸出対象備品。備品名、総在庫数、対象部署を持つ |
| `equipment_loan_requests` | 作成 / 更新 / 参照 | 貸出申請トランザクション。申請者、備品、ステータス、貸出期間、理由を持つ |

### 6.3 カラム要件

| テーブル | カラム | 型 | 必須 | 概要 |
|----------|--------|----|------|------|
| `users` | `id` | integer | 必須 | 主キー |
| `users` | `name` | string | 必須 | 氏名 |
| `users` | `department` | string | 必須 | 部署名 |
| `users` | `role` | enum/string | 必須 | `admin` または `staff` |
| `equipments` | `id` | integer | 必須 | 主キー |
| `equipments` | `name` | string | 必須 | 備品名 |
| `equipments` | `stock_count` | integer | 必須 | 総在庫数 |
| `equipments` | `target_department` | string/null | 任意 | NULL の場合は全部署に貸出可能 |
| `equipment_loan_requests` | `id` | integer | 必須 | 主キー |
| `equipment_loan_requests` | `user_id` | foreign key | 必須 | `users.id` |
| `equipment_loan_requests` | `equipment_id` | foreign key | 必須 | `equipments.id` |
| `equipment_loan_requests` | `status` | enum/string | 必須 | `pending` / `approved` / `returned` / `rejected` |
| `equipment_loan_requests` | `requested_from` | date | 必須 | 貸出開始日 |
| `equipment_loan_requests` | `requested_to` | date | 必須 | 貸出終了日 |
| `equipment_loan_requests` | `reason` | text/null | 任意 | 申請理由 |

返却期限超過は DB カラムとして保持せず、`status = approved` かつ `requested_to < 今日` の条件で算出する。

## 7. バリデーション・エラー

| 条件 | エラーコード | メッセージ |
|------|-------------|-----------|
| `mock_user_id` が未指定 | 422 | 操作ユーザーを選択してください |
| `mock_user_id` が存在しない | 404 | 操作ユーザーが見つかりません |
| `equipment_id` が未指定 | 422 | 備品を選択してください |
| `equipment_id` が存在しない | 404 | 備品が見つかりません |
| `requested_from` が未指定 | 422 | 貸出開始日を入力してください |
| `requested_to` が未指定 | 422 | 貸出終了日を入力してください |
| `requested_to` が `requested_from` より前 | 422 | 貸出終了日は貸出開始日以降の日付を入力してください |
| `requested_from` が過去日 | 422 | 貸出開始日は本日以降の日付を入力してください |
| 備品の `target_department` が NULL ではなく、申請ユーザーの `department` と一致しない | 403 | この備品は所属部署では申請できません |
| 同一ユーザーが、重複する期間に別の備品を `pending` または `approved` で申請済み | 422 | 指定期間は既に別の備品を申請中または貸出中です |
| 同一備品の対象期間における `approved` 件数が `stock_count` 以上 | 422 | 指定期間は備品の在庫数を超えています |
| 一般社員がステータス更新を行う | 403 | ステータスを更新する権限がありません |
| `status` が `approved` / `rejected` / `returned` 以外 | 422 | 指定されたステータスは更新できません |

重複期間の判定は、既存申請の `requested_from` から `requested_to` と新規申請期間が1日でも重なる場合を対象とする。既存申請のうち `returned` / `rejected` は重複判定対象外とする。
貸出開始日と貸出終了日が同日の申請は許可する。ステータス遷移の順序制約は設けず、管理者は許可されたステータスへ直接更新できる。

## 8. 権限

| ロール | 操作可否 |
|--------|----------|
| `staff` | 自分の申請一覧のみ取得可能。新規申請可能。ステータス更新不可 |
| `admin` | 全ユーザーの申請一覧を取得可能。新規申請可能。ステータスを `approved` / `rejected` / `returned` に更新可能 |

## 9. 非機能要件

- `GET /api/equipment-loans` は N+1 問題を避けるため、`user` と `equipment` のリレーションを Eager Loading する。
- DB は既存 Docker の PostgreSQL 18 を使用する。
- PoC のためパスワード認証は実装しない。ただし、API は `mock_user_id` に基づく権限チェックを必ず実施する。
- E2E は Playwright を主テストとする。Service ロジックが分離される場合は PHPUnit、JavaScript ロジックが分離される場合は Vitest を併用する。
- 初期データは Seeder で投入する。具体的なユーザー名、部署、備品名はフェーズ 2 で設計する。

## 10. 具体例（入出力サンプル）

### 正常系の例

前提:

- ユーザー: 山田 太郎（開発部、`staff`）
- 備品: MacBook Pro M3（`target_department = 開発部`）
- 山田 太郎には 2026-07-10 から 2026-07-12 に重複する `pending` / `approved` 申請がない

入力:

- `mock_user_id`: 山田 太郎の ID
- `equipment_id`: MacBook Pro M3 の ID
- `requested_from`: `2026-07-10`
- `requested_to`: `2026-07-12`
- `reason`: `検証作業で使用するため`

期待結果:

- `equipment_loan_requests` に `status = pending` の申請が登録される。
- 一覧に新規申請が表示される。

### 異常系の例

前提:

- ユーザー: 佐藤 花子（総務部、`staff`）
- 備品: MacBook Pro M3（`target_department = 開発部`）

入力:

- `mock_user_id`: 佐藤 花子の ID
- `equipment_id`: MacBook Pro M3 の ID
- `requested_from`: `2026-07-10`
- `requested_to`: `2026-07-12`

期待結果:

- HTTP 403 相当のエラーとなる。
- 申請は登録されない。
- 画面に「この備品は所属部署では申請できません」と表示される。

## 11. 受け入れ条件

- [ ] 一般社員を選択して一覧を表示すると、そのユーザー自身の申請のみが表示され、管理者用のステータス更新ボタンは表示されない。
- [ ] 管理者を選択して一覧を表示すると、全ユーザーの申請が表示され、各行に「承認」「却下」「返却」ボタンが表示される。
- [ ] 備品名検索に文字列を入力すると、備品名が部分一致する申請のみが一覧に表示される。
- [ ] 一般社員を選択して返却期限超過の貸出中申請がある場合、画面上部に自分の期限切れアラートが表示され、対象行に注意アイコンが表示される。
- [ ] 管理者を選択して返却期限超過の貸出中申請がある場合、画面上部に全ユーザー分の期限切れアラートが表示され、対象行に注意アイコンが表示される。
- [ ] 返却期限超過の申請がない場合、画面上部の期限切れアラートと一覧行の注意アイコンは表示されない。
- [ ] 部署対象が一致する備品を一般社員が申請すると、`pending` の申請が登録され、一覧に表示される。
- [ ] 貸出開始日が過去日の申請は登録されず、画面にエラーが表示される。
- [ ] 部署対象が一致しない備品を一般社員が申請すると、申請は登録されず、画面にエラーが表示される。
- [ ] 同一ユーザーが重複期間に別備品を申請すると、申請は登録されず、画面にエラーが表示される。
- [ ] 同一備品の同時貸出数が `stock_count` を超える申請は登録されず、画面にエラーが表示される。
- [ ] 管理者が「承認」「却下」「返却」を操作すると、対象申請のステータスが更新され、一覧のバッジ表示に反映される。
- [ ] 一般社員がステータス更新 API を呼び出した場合、HTTP 403 相当のエラーとなり、ステータスは更新されない。
