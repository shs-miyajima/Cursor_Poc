# テスト計画 — アンケート作成・回答・管理システム

> status: 03-test-plan.status を参照
> 前提: 02-design.status が `approved`

## 1. テスト方針

| 種別 | 対象 | ツール | 備考 |
|------|------|--------|------|
| 単体（サーバー） | `AuthService` / `UserCsvImportService` / `SurveyService` / `SurveyAnswerService` / `SurveyResultService` / `Survey::effectiveStatus` / `Gender` enum / User モデル（ハッシュ・小文字化） | PHPUnit | DB を使う Feature テスト形式（`RefreshDatabase`） |
| 単体（フロント） | `modules/surveyFormState.js` / `modules/chartDataTransformer.js`（純関数） | Vitest | DOM 非依存 |
| E2E | 画面・権限・遷移・バリデーション表示 | Playwright（TypeScript） | 主テスト。並列実行前提 |

- E2E で検証困難な観点（パスワードハッシュの保存形式、回答 1,000 件の集計性能、年代境界の網羅）は
  PHPUnit で代替する（要件カバレッジ表の備考に明記）。
- バリデーションは E2E で「画面での文言表示 + データ不変」を、PHPUnit で「Service 層の例外・戻り値」を確認する。

## 2. テスト種別別・カテゴリ別件数

CSV はテスト種別ごとに分けて作成する。E2E / PHPUnit / Vitest のケースを同じ CSV に混在させない。

## 3. 網羅基準

- [x] 正常系（代表パターン）
- [x] 異常系（バリデーション、存在しない ID 等）
- [x] 境界値（100/101 文字、7/8 文字、255/256 文字、500/501 文字・行、選択肢 1/2/10/11 個、設問 0/50/51 問、回答 0 件、1000/1001 文字）
- [x] 権限パターン（3 ロール × 全操作の可否、他社データアクセス、全体ビュー/企業ビュー）
- [x] 派生パターン（状態遷移: 下書き→公開→終了・締切自動終了、CSV 新規/更新、絞り込みあり/なし/不一致）
- [x] 更新系の空入力（必須: E2E-022/045/060/169/170、任意: E2E-047/049/050/171/172、例外仕様のパスワード: E2E-046）
- [x] CSV の各行が 1 ケース 1 観点になっていること
- [x] OK ケースと NG ケースの対（例: パスワード 7 文字 NG / 8 文字 OK、必須設問 ON/OFF、絞り込み一致/不一致/指定なし）
- [x] 要件で許可されている操作・状態遷移を全て列挙（公開・終了・削除・編集・回答・修正・企業ビュー切替/解除）
- [x] 条件分岐の取り得る値ごとのケース（絞り込み指定なし: E2E-133、締切なし: PU-026、企業コード空欄: E2E-001/PU-007）
- [x] 要件カバレッジ表を作成し、対応漏れがないこと（NFR-xx 含む）
- [x] 期待結果は具体的な値・文言・件数で記載（曖昧語なし）
- [x] 独立性・再現性（下記）を満たしていること
- [x] `02-design.md` §5 の「影響あり」項目に回帰確認ケースが対応していること（下記「回帰確認」）

### 期待結果の書き方（曖昧表現の排除）

各 CSV の期待結果は文言・件数・遷移先 URL・画面状態で固定した（「正しく」「適切に」等は不使用）。
要件にエラーメッセージ文言が未定義の箇所（ユーザー氏名の最大長 = E2E-042、パスワード上限 = E2E-178、
CSV エラーの理由文言 = E2E-074/076〜078/180/182）は、要件 §7 の同型式文言をテスト計画側で
固定値として定めた（備考に明記）。

グラフ描画内容（E2E-129/130/144）は Chart.js の canvas 描画のため DOM の文言アサートでは検証できない。
E2E では「回答件数の DOM 表示」+「チャートに紐づく集計データ（canvas の data 属性または
`Chart.getChart()` の data）」を検証し、データ→描画の変換の正しさは VT-006〜VT-008 で担保する。
このためダッシュボード実装時に canvas 要素へ検証用のデータ属性を付与する（実装フェーズへの申し送り。
挙動・画面仕様への影響はないため 02-design.md の差分承認は不要と判断、承認確認に明記）。

### 独立性・再現性の標準観点

- [x] 各 E2E ケースは前提条件に「本ケース専用の企業（一意な企業コード）を新規作成」を明記し、
      他ケースが作成したデータを前提にしない
- [x] 共有データは Seeder のスーパーユーザーアカウント（su@example.com）のみ。
      参照（ログイン）だけに使い、更新・削除するケースはない
- [x] 実行順序を入れ替えても成立する（ケース間のデータ参照なし）
- [x] PHPUnit は `RefreshDatabase`、Vitest は純関数のため状態を持たない

#### 並列実行安全性（E2E は並列実行が前提）

- [x] 全ケースがケース内で専用企業（企業コードに Test ID またはランダム値を含める）を作成して
      操作するため、レコードの奪い合いが発生しない
- [x] 全体ビューのサマリ件数（E2E-026）は全企業合計で並列実行の影響を受けるため、
      絶対件数ではなく「作成した企業が一覧に含まれること」を検証する（CSV 備考に明記）
- [x] Seeder 初期データ（スーパーユーザー）を更新・削除するケースはない
