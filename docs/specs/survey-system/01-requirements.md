# 要件定義 — アンケート作成・回答・管理システム

> status: 01-requirements.status を参照（`draft` | `approved` | `rejected`）
>
> `open-questions.md` の全質問（5 回・計 45 問）の回答を反映した確定案。仮定なし。

## 1. 概要

企業向けのアンケート作成・回答・管理システムを新規に構築する。
企業の管理者（人事部等）がアンケートを作成・公開し、企業の社員（ユーザー）が回答する。
回答結果は管理者がダッシュボードでグラフ形式（縦棒グラフ／円グラフ）で確認・絞り込みできる。
弊社社員はスーパーユーザーとして企業・管理者・ユーザーの登録と、企業に代わった
アンケート作成を行える。

実行基盤は既存のマルチコンテナ構成（app: PHP 8.4 / Laravel 12、nginx、db: PostgreSQL 18）を
そのまま使用する。言語は Docker イメージに含まれる PHP（バックエンド）と
JavaScript（フロント、Vite + Tailwind CSS）とする。

## 2. 用語集

| 用語 | 定義 |
|------|------|
| スーパーユーザー | 弊社社員。全企業のデータにアクセスできる最上位ロール。特定の企業には属さない（company_id なし） |
| 管理者 | 企業の人事部など、企業内でアンケート管理・ユーザー管理・部署管理・結果確認を行うロール。自社のデータのみ扱える |
| ユーザー | 企業の社員。自社に公開されたアンケートへの回答のみ行える。アンケートに回答できるのはユーザーロールのみ |
| 企業 | 管理者・ユーザーが所属する組織単位。アンケートは企業単位で作成・公開される |
| 企業コード | 企業を識別する半角英数字のコード。企業登録時にスーパーユーザーが入力し、全企業で一意。ログイン時の企業特定に使用する |
| 部署 | 企業内の組織単位。部署マスタ（名称のみ）として管理者が登録・編集・削除する。ユーザーの属性およびダッシュボード絞り込みに使用する |
| アンケート | 1 つ以上の設問で構成される調査。回答対象は作成した企業の全ユーザー |
| 設問 | アンケート内の 1 問。形式は単一選択・複数選択・自由記述の 3 種 |
| 回答 | 1 ユーザーによる 1 アンケートへの提出。公開中は何度でも修正（再提出）できる。記名式 |
| アンケート状態 | `下書き` → `公開` → `終了` の 3 状態。手動切替に加え、締切日時の到達で自動的に終了扱いになる |
| 論理削除 | データを DB に残したまま非表示・利用不可にする削除方式。本システムの削除はすべて論理削除 |
| 全体ビュー | スーパーユーザー専用の全企業横断画面 |
| 企業ビュー | スーパーユーザーが特定の企業を選択して表示する画面。管理者が見る画面と同等の内容 |

## 3. スコープ

### 対象

- [x] Laravel 管理画面（認証・企業/ユーザー/部署管理・アンケート CRUD・状態管理・回答・ダッシュボード）
- [x] フロント（Blade / Vite / JavaScript / Tailwind CSS、Chart.js によるグラフ描画）
- [x] DB スキーマ変更（新規テーブル追加、users テーブル拡張）

### 対象外

- 既存の `welcome` 画面（トップページ）の変更は、ログイン画面へのリンク追加のみ。他は変更しない
- メール送信（招待メール・通知メール等）。パスワードは登録・編集時に登録者が直接設定する
- 本人によるパスワードリセット（管理者・スーパーユーザーによる再設定で代替）
- アンケート回答の途中保存（下書き保存はアンケート作成側の状態管理でのみ提供）
- 配信先の絞り込み（部署指定等）。回答対象は常に自社の全ユーザー
- 外部システム連携・API 公開
- sample-crud など既存 spec の資産

#### S-05 ユーザー CSV 登録

| 項目名 | 型 | 必須 | 最大長 | 初期値 | 表示条件・備考 |
|--------|----|------|--------|--------|----------------|
| CSV ファイル | file | ○ | 2MB・データ行 500 行 | - | UTF-8、ヘッダー行あり。列: 氏名, メールアドレス, パスワード, 部署, 性別, 生年月日, 入社年月。登録されるロールはユーザー固定（管理者は個別画面で登録する） |
| エラー一覧 | 表示 | - | - | - | 不正行がある場合のみ。全行チェック後、「N 行目: 項目名 — 理由」を全件表示（1 件目で中断しない）。この場合は確認画面に進まない |

CSV 列の形式:

| 列 | 必須 | 形式 |
|----|------|------|
| 氏名 | ○ | 100 文字以内 |
| メールアドレス | ○ | メール形式。小文字に正規化して保存・比較する（`Taro@…` と `taro@…` は同一とみなす） |
| パスワード | ○ | 8〜255 文字 |
| 部署 | △ | 自社の部署マスタに存在する部署名。照合は英字の大文字小文字を区別しない（`営業G` = `営業g`。マスタ側の表記で紐付け）。全角半角は区別する。存在しない名称はエラー行（自動登録しない） |
| 性別 | △ | 「男性」「女性」「その他」「未回答」のいずれかの文字列。空欄は「未回答」 |
| 生年月日 | △ | `YYYY-MM-DD` |
| 入社年月 | △ | `YYYY-MM` |

取込みの流れ（新規／更新の判定）:

1. アップロード時に全行をチェックする。エラーが 1 件でもあればエラー一覧のみを表示し、登録・更新は行わない
2. エラーが 0 件の場合、確認画面（S-05a）に進む。メールアドレス（小文字正規化後）が自社の既存ユーザー
   （論理削除済みを除く）と一致する行は「更新」、一致しない行は「新規」と区分表示する
3. 「確定」で新規行はユーザー登録、更新行は既存ユーザーの氏名・パスワード・部署・性別・生年月日・
   入社年月を CSV の値で上書きする（更新行の任意列が空欄の場合も空値で上書きする）。
   確定せずに「戻る」またはページを離れた場合は何も反映しない
4. 更新対象になるのはユーザーロールのみ。既存の管理者と同じメールアドレスの行はエラー行として扱う

## 6. データ要件

| テーブル | 操作 | 概要 |
|----------|------|------|
| companies | CRUD | 企業（id, 企業名, 企業コード, deleted_at, タイムスタンプ）。削除は論理削除（deleted_at） |
| departments | CRUD | 部署マスタ（id, company_id, 部署名, deleted_at, タイムスタンプ）。同一企業内で名称一意（論理削除済みを除く） |
| users | CRUD | 既存テーブルを拡張: company_id（スーパーユーザーは NULL）, role（superuser/admin/user）, department_id（任意）, 性別, 生年月日, 入社年月, deleted_at を追加 |
| surveys | CRUD | アンケート（id, company_id, タイトル, 説明, 状態[draft/published/closed], 締切日時, 作成者 user_id, deleted_at, タイムスタンプ） |
| questions | CRUD | 設問（id, survey_id, 設問文, 形式, 必須フラグ, 表示順）。下書き中のみ変更可 |
| question_options | CRUD | 選択肢（id, question_id, ラベル, 表示順）。下書き中のみ変更可 |
| survey_responses | C/R/U | 回答ヘッダ（id, survey_id, user_id, 提出日時, 最終更新日時）。survey_id + user_id で一意。修正時は同一レコードを更新 |
| survey_answers | CRUD | 回答明細（id, survey_response_id, question_id, question_option_id または自由記述テキスト）。修正時は洗い替え（delete + insert） |

- 一意制約: 企業名・企業コードは全企業で一意、メールアドレスは同一企業内で一意
  （スーパーユーザーは company_id NULL 同士で一意）、部署名は同一企業内で一意。いずれも論理削除済みを除く
- メールアドレスは小文字に正規化して保存し、一意判定・ログイン照合・CSV の新規/更新判定も
  正規化後の値で行う。部署名のマスタ照合は英字の大文字小文字を区別しない（全角半角は区別する）
- 削除はすべて論理削除。論理削除された企業に所属する管理者・ユーザーはログイン不可。
  論理削除されたユーザーの過去回答は集計に残る。論理削除された部署の所属ユーザーは「部署未設定」扱い
- 絞り込みの「回答日時」は survey_responses の最終更新日時を用いる
- スーパーユーザーの初期アカウントは seeder で投入する
- 締切日時による自動終了は、バッチではなく参照時に「状態 = 公開 かつ 締切日時 < 現在時刻」を
  終了とみなす方式とする（新規コンテナ・スケジューラ設定を追加しないため）

## 10. 具体例（入出力サンプル）

### 正常系の例

1. スーパーユーザーが企業「株式会社サンプル」（企業コード: `sample01`）を登録し、
   管理者 admin@example-corp.jp を登録する。
2. 管理者が企業コード `sample01` + メール + パスワードでログイン → ダッシュボードが表示される。
3. 管理者が部署「営業部」「総務部」を登録し、ユーザー taro@example-corp.jp（部署: 営業部）を登録する。
4. 管理者が S-07 で以下を入力し「公開」を押す。
   - タイトル: 「社内満足度調査 2026」／締切日時: 2026-07-31 23:59
   - 設問 1: 「現在の職場環境に満足していますか」/ 単一選択 / 選択肢「満足・普通・不満」/ 必須 ON
   - 設問 2: 「改善してほしい点を教えてください」/ 自由記述 / 必須 OFF
5. ユーザー taro が `sample01` でログイン → S-09 未回答一覧に表示される。
   設問 1 で「満足」を選択し送信 → 「回答を送信しました」と表示され、回答済に移動する。
6. 同ユーザーが再度開くと「満足」が選択済みで表示され、「普通」に変更して「回答を修正する」で再送信 → 上書き保存される。
7. 管理者が S-08 で当該アンケートを選択 → 設問 1 の縦棒グラフに「普通: 1 件」、回答件数「1 件」と表示される。
   円グラフに切り替えると同じ集計が円グラフで表示される。
8. 絞り込みで部署 =「営業部」を指定 → 回答件数「1 件」のまま。部署 =「総務部」を指定 → 回答件数「0 件」になる。
9. 別企業（企業コード: `other01`）にも taro@example-corp.jp と同じメールアドレスのユーザーを登録でき、
   それぞれの企業コードでログインできる。

### 異常系の例

1. ユーザーが設問 1（必須）を未選択のまま送信 → 「この設問は必須です」が設問 1 の下に表示され、保存されない（VAL-28）。
2. 管理者が当該アンケートを「終了」に切替後、ユーザーが回答を修正して送信 → 「このアンケートは回答を受け付けていません」が表示され、保存されない（VAL-30）。
3. 管理者が CSV 登録で 3 行目のメールアドレスがファイル内の 1 行目と重複し、5 行目の部署「企画部」が
   部署マスタに存在しないファイル（10 行）をアップロード → 全行チェック後、
   「3 行目: メールアドレス — ファイル内で重複しています」「5 行目: 部署 — 部署マスタに登録されていません」が
   まとめて表示され、確認画面に進まず 1 件も登録・更新されない（VAL-18）。
4. 管理者が他社アンケートの結果 URL（`/api/surveys/{他社id}/results`）を直接開く → 403 が返る（NFR-04）。
5. ログインで存在しない企業コード `wrong99` を入力 → 「ログイン情報が正しくありません」が表示される（VAL-01）。

## 11. 受け入れ条件

| ID | 受け入れ条件 |
|----|-------------|
| AC-01 | スーパーユーザーが企業コード空欄 + メールアドレス・パスワードでログインすると全体ビュー（S-03）が表示される |
| AC-02 | 管理者が企業コード + メール + パスワードでログインするとダッシュボード（S-08）が、ユーザーがログインするとアンケート一覧（S-09）が表示される |
| AC-03 | 誤ったパスワードまたは誤った企業コードでログインすると「ログイン情報が正しくありません」が表示され、ログインできない |
| AC-04 | スーパーユーザーが企業名・企業コードを入力して登録すると、企業一覧に追加され、ユーザー登録画面の企業選択に表示される |
| AC-05 | スーパーユーザーが企業名を編集して保存すると、企業一覧の表示が新しい名称に変わる |
| AC-06 | スーパーユーザーが企業を削除すると、企業一覧から消え、所属していた管理者・ユーザーはログインできなくなる |
| AC-07 | スーパーユーザーが企業とロール（管理者）を指定してアカウントを登録すると、そのアカウントで管理者としてログインできる |
| AC-08 | スーパーユーザーが企業ビューに切り替えると、選択した企業のダッシュボード・アンケート一覧・ユーザー一覧・部署一覧が管理者と同じ内容で表示される |
| AC-09 | スーパーユーザーが企業を指定してアンケートを作成・公開すると、その企業のユーザーの未回答一覧に表示される |
| AC-10 | 管理者が個別画面でユーザーを登録すると、そのアカウントでユーザーとしてログインできる |
| AC-11 | 管理者がロール「管理者」を指定して登録すると、そのアカウントで自社の管理者としてログインできる |
| AC-12 | 異なる企業に同一メールアドレスのユーザーを登録でき、それぞれの企業コードでログインできる |
| AC-13 | 管理者がユーザーの部署を編集して保存すると、ユーザー一覧とダッシュボードの部署絞り込みに反映される |
| AC-14 | 管理者がユーザーを削除すると、そのアカウントはログインできず一覧に表示されないが、過去の回答は集計に残る |
| AC-15 | 管理者が部署を登録すると、ユーザー登録・編集の部署選択肢とダッシュボードの部署絞り込みに表示される |
| AC-16 | 管理者が部署を削除すると、部署選択肢から消え、所属していたユーザーの部署表示は「未設定」になる |
| AC-17 | 管理者が正常な CSV（複数行・全行新規）をアップロードすると、確認画面に全行が「新規」と表示され、「確定」で全行がユーザー一覧に追加される |
| AC-18 | 管理者が不正行を含む CSV をアップロードすると、全行チェック後に行番号・項目・理由の一覧が表示され、確認画面に進まず 1 件も登録・更新されない |
| AC-19 | 管理者がタイトルと設問を入力して「下書き保存」すると、管理一覧に「下書き」で表示され、ユーザーの一覧には表示されない |
| AC-20 | 管理者が下書きアンケートの設問を編集して「公開」すると、編集後の内容が自社ユーザーの未回答一覧に表示される |
| AC-21 | 管理者が公開中アンケートを「終了」に切り替えると、ユーザーは回答・修正できなくなり、読み取り専用表示になる |
| AC-22 | 締切日時を過ぎたアンケートは、ユーザーが開いても回答フォームが表示されず回答できない |
| AC-23 | 管理者がアンケートを削除すると、ユーザーの一覧とダッシュボードのアンケート選択肢から消える |
| AC-24 | ユーザーが未回答アンケートに回答して送信すると、回答済一覧に移動する |
| AC-25 | ユーザーが公開中の回答済アンケートを開くと前回の回答が初期表示され、「回答を修正する」で再送信すると最新の回答が集計に反映される |
| AC-26 | 管理者がダッシュボードでアンケートを選択すると、設問ごとの回答集計が縦棒グラフで表示され、回答件数が表示される |
| AC-27 | 管理者がグラフ形式を円グラフに切り替えると、同じ集計データが円グラフで再描画される |
| AC-28 | 管理者が絞り込み条件（部署・回答日時・性別・年代・入社年月のいずれか）を指定すると、条件に合致する回答のみが集計・グラフに反映され、件数表示も更新される |
| AC-29 | 管理者が他社のアンケート結果・ユーザー一覧・部署一覧を URL 直指定で開くと 403 または 404 が表示される |
| AC-30 | ユーザーが管理系 URL（`/dashboard` 等）を直接開くと 403 が表示される |
| AC-31 | 既存ユーザーと同じメールアドレスの行を含む CSV をアップロードすると、確認画面で該当行が「更新」、それ以外が「新規」と区分表示され、件数サマリ（新規 N 件・更新 M 件）が表示される |
| AC-32 | 確認画面で「確定」すると更新行の既存ユーザー情報（氏名・部署・属性等）が CSV の値で上書きされ、「戻る」を押した場合は何も反映されない |
| AC-33 | 大文字を含むメールアドレス（例: Taro@example-corp.jp）で登録しても小文字（taro@example-corp.jp）で保存され、どちらの表記でもログインでき、CSV では同一ユーザー（更新）と判定される |
