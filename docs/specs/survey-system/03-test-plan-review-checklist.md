# テスト計画レビューチェックリスト（AI 独立レビュー）— アンケート作成・回答・管理システム

> 目的: `03-test-plan.md` §3 の「網羅基準」は**作成者（AI）のセルフチェック**であり、
> 作成者自身の見落としには気づけない。作成コンテキストを引き継がない**専用レビュー
> サブエージェント**（`sdd-plan-reviewer`、readonly）が、このチェックリストに沿って別視点で検証する。
>
> 運用: `.cursor/rules/sdd-workflow.mdc`「フェーズ 3 > 独立レビュー（承認確認前に必須）」を参照。
> このレビューは人間の承認を代替しない（自己承認の禁止の原則は変わらない）。

## 使い方

1. `03-test-plan.md` と各 CSV の作成後、**承認確認を人間に提示する前**に、専用レビューサブエージェント
   `sdd-plan-reviewer`（定義: `.cursor/agents/sdd-plan-reviewer.md`、readonly）へ
   `01-requirements.md` / `02-design.md` / `03-test-plan.md` / 各 CSV / 本チェックリストのパスを渡し、
   下記項目を ✓ / ✕ と理由で判定させる
2. ✕ の指摘のうち機械的に自動修正可能なもの（カバレッジ表の空欄、曖昧な期待結果の言い換え、
   出典 ID の記載漏れ、CSV 粒度の分割漏れ等）はメインの AI がその場で修正する
3. 業務ルールの解釈・仕様追加の要否など判断が必要な指摘は自動修正せず、承認確認の提示に
   含めて人間に判断を仰ぐ
4. 判定結果と対応状況を「6. 総合判定」の実施記録に記入する

## 1. トレーサビリティ

- [x] 要件カバレッジ表に `01-requirements.md` の UC / VAL（§7）/ AC（§11）/ NFR（§9）/
      画面項目（§5.2）/ 権限（§8）の全項目が記載されているか（記載漏れがないか）
- [x] カバレッジ表の「対応 Test ID」が空欄の行がないか（あれば意図的な対象外か確認する）
      （NFR-08 のみ「手動確認」で理由明記済み）
- [x] NFR-xx が「対応不可」の場合、理由と代替手段（PHPUnit・手動確認等）が明記されているか
      （NFR-01/06 = PHPUnit 代替、NFR-08 = フェーズ 4 完了報告での手動確認）
- [x] `02-design.md` §5 で「影響あり」とした `IMPACT-xx` に、`03-test-plan.md`「回帰確認」で
      対応する Test ID が割り当てられているか（IMPACT-01/02/03/05/06/07/10 の 7 件すべて対応）
- [x] CSV「出典」列の要件 ID が `01-requirements.md` に実在するか（lint で機械検証済み・ERROR 0）
- [x] 逆方向（Test ID → 要件 ID）にも意味不明な出典がないか

## 2. 期待結果の質

- [x] 期待結果が具体的な値・文言・件数・画面状態で書かれているか
      （初回 ✕: E2E-089 の期待結果が「読み取り専用または受付終了表示」の選言 → 前提を未回答に固定し
      「回答受付は終了しました」に一意化して解消）
- [x] エラーケースでエラーメッセージの文言まで固定されているか
      （要件未定義の文言 = E2E-042/074/076〜078/178/180/182 は計画側で固定値を定め備考に明記）
- [x] 同じ観点のケース間で期待結果の記述粒度が大きくブレていないか

## 3. ケース粒度・独立性

- [x] 1 行 1 観点になっているか
      （初回 ✕ 4 件: E2E-134（日時範囲の内/外）→ E2E-134/183 に分割、E2E-053（3 属性）→
      E2E-053/184/185 に分割、PU-032 → PU-032/042 に分割、PU-035 → PU-035/043 に分割）
- [x] 前提条件が単独で再現可能な記載か
      （初回 ✕: 36 ケースが「管理者でログイン済み」のみで専用企業の作成が不明 →
      「本ケース専用の企業と管理者を新規作成しログイン済み」に統一して解消）
- [x] 複数ケースが同一データを書き換える場合、テストデータが分離されているか（ケースごと専用企業）
- [x] 実行順序への依存がないか
- [x] 並列実行安全性: 共有データは Seeder の SU アカウントのみで参照専用。
      全体ビューサマリ（E2E-026）は絶対件数でなく包含検証に変更済み

## 4. 網羅性

- [x] 正常系・異常系が対になっているか
- [x] 境界値の両側（ちょうど / 超過）が確認されているか
      （初回 ✕ 7 項目: 説明 1000・設問文 500・選択肢 100・部署名 100・パスワード 255/256・
      CSV パスワード 255/256・CSV 氏名 100/101 → E2E-173〜182 を追加して解消）
- [x] 全ロール × 全操作の権限パターンが確認されているか
      （初回 ✕ 6 組: ユーザーロールの編集・削除・状態切替系 URL、SU 全体ビューの部署画面 →
      E2E-186〜191 を追加して解消）
- [x] 要件で許可されている操作・状態遷移が全て列挙されているか
      （初回 ✕: S-06 一覧の「公開」ボタン実行ケースなし → E2E-192 を追加して解消）
- [x] 更新系の空入力ケースが全項目分あるか
      （初回 ✕ 4 項目: ユーザー編集メール・アンケート編集タイトル/説明/締切 →
      E2E-169〜172 を追加して解消）

## 5. 実装可能性（先読みリスク）

- [x] Playwright で実際に自動化できる内容か
      （初回 ✕【要判断】: E2E-089 の締切実時間待機 → 締切=現在+1 分で公開し表示変化を
      ポーリング（上限 90 秒）する方式に確定。ロジック網羅は PU-024〜026 で担保。承認確認に明記）
- [x] PHPUnit / Vitest の担保先が設計書のクラス・関数と対応しているか（§6/§7 の対応表で明示）
- [x] 固定時間の待機に頼った書き方を前提にしていないか
      （初回 ✕【要判断】: E2E-129/130/144 のグラフ描画内容は canvas のため DOM アサート不可 →
      canvas のデータ属性 + 回答件数 DOM 表示の検証に変更し、変換ロジックは VT-006〜008 で担保。
      実装時に検証用データ属性を付与する（画面仕様への影響なし）。承認確認に明記）

## 6. 総合判定

| 状態 | 条件 | 対応 |
|------|------|------|
| 指摘なし | 1〜5 のすべての項目が ✓ | そのまま承認確認を人間に提示する |
| 自動修正済み | ✕ はあったが機械的に修正できた | 修正内容を記載の上、承認確認を人間に提示する |
| 要判断 | 業務ルールの解釈等、自動修正できない ✕ が残る | 承認確認に残課題として明記し、人間に判断を仰ぐ（自己判断で解消したことにしない） |
